# تحليل تقني لمشروع YallaSakn

## نظرة عامة سريعة
- المشروع مبني بـ **React + TypeScript + Vite** مع استخدام **TanStack Query** و**React Router** و**i18next**.
- البنية الحالية تسمح بالتوسع بسرعة، لكن يوجد دين تقني واضح في الاتساق بين الملفات والجودة النوعية (Typing/Lint/Test).

## نقاط القوة
1. **Stack حديث ومناسب للمنتجات السريعة**: Vite + React 18 + TypeScript.
2. **وجود حدود أخطاء على مستوى التطبيق** (Error Boundary) يقلل انهيار واجهة المستخدم بالكامل عند حدوث أخطاء runtime.
3. **فصل معقول لبعض المنطق في hooks** خصوصًا في صفحة البحث.
4. **تعدد لغات (عربي/إنجليزي)** موجود ومفعّل عبر i18next.

## نقاط الضعف والمخاطر الحالية

### 1) جودة الكود (Lint) منخفضة حاليًا
- فحص `eslint` يعرض عددًا كبيرًا جدًا من المشاكل (231 error, 15 warning).
- كثرة `any` تعني فقدان فوائد TypeScript واحتمالية أخطاء runtime.
- توجد ملفات بها تنسيق/محارف غير قياسية (irregular whitespace) تسبب فشل lint بالكامل.

**الأثر**: صعوبة الصيانة، زيادة bugs، تأخير أي تطوير جديد.

### 2) عدم اتساق معماري ووجود نسخ مكررة/غير مستخدمة
- وجود ملف `src/App.tsx.backup` يحتوي محتوى مكرر ومكسور.
- وجود مسارين لمنطق البحث (`src/pages/Search.tsx` و`src/pages/Search/index.tsx`) بواجهات مختلفة، ما يربك الفريق.
- وجود نسختين من `useFormState` بامتدادين مختلفين (`.ts` و`.tsx`).

**الأثر**: زيادة الالتباس واحتمال تعديل ملف غير مستخدم عن طريق الخطأ.

### 3) طبقة API غير موحدة
- يوجد أكثر من `axiosInstance` بقيم baseURL مختلفة (إحداها hardcoded localhost).
- بعض الأماكن تستخدم `fetch` مباشرة بدل عميل API موحد.

**الأثر**: سلوك مختلف بين البيئات، صعوبة إدارة auth/interceptors/headers.

### 4) أداء الواجهة (حجم الباندل)
- مخرجات البناء تُظهر bundle JavaScript كبير (~979KB قبل gzip) وتحذير عن chunks > 500KB.

**الأثر**: تحميل أبطأ خاصة على الشبكات الضعيفة/الموبايل.

### 5) غياب اختبارات تلقائية
- لا توجد اختبارات unit/integration ظاهرة في المستودع.

**الأثر**: أي refactor عالي المخاطر، وضعف الثقة في النشر.

## خطة تحسين عملية (مقترح تنفيذ على مراحل)

## المرحلة 1 (سريعة: 1–3 أيام)
1. تصفير مشاكل lint الحرجة تدريجيًا:
   - إزالة المحارف غير القياسية.
   - استبدال `any` بالأنواع الأساسية أو `unknown` مع type guards.
2. تنظيف الملفات غير المستخدمة:
   - حذف `App.tsx.backup`.
   - توحيد نقطة الدخول لصفحة البحث (اختيار ملف واحد فقط).
   - حذف النسخ المكررة من hooks.
3. توحيد API client:
   - ملف axios واحد فقط + baseURL من env.

## المرحلة 2 (متوسطة: 3–7 أيام)
1. إدخال **React Query** بشكل أوسع للـ fetching/caching بدل `fetch` المباشر.
2. تقسيم الشاشات الثقيلة lazy-loading عبر `React.lazy` و`Suspense`.
3. وضع قواعد typing موحدة لأنواع API responses وDTO mapping.

## المرحلة 3 (تحسين الجودة والاستدامة)
1. إضافة اختبارات:
   - Vitest + Testing Library لسيناريوهات hooks والمكونات الأساسية.
   - Playwright smoke tests للمسارات الحرجة (login/search/details).
2. CI pipeline:
   - `lint + build + test` على كل PR.
3. قياس الأداء (Lighthouse/Web Vitals) قبل/بعد التحسين.

## أولويات تنفيذ مباشرة (Top 5)
1. حل مشاكل `lint` التي تمنع الجودة baseline.
2. توحيد ملف/مسار البحث وإزالة الملفات المكررة.
3. توحيد `axiosInstance` ومنع hardcoded localhost.
4. تقليل حجم bundle عبر code-splitting.
5. إضافة مجموعة اختبارات حد أدنى + CI.

## تقييم القوة الحالية
- **الأساس التقني**: جيد جدًا (8/10).
- **القابلية للصيانة**: متوسطة إلى ضعيفة حاليًا بسبب الدين التقني (4/10).
- **الجاهزية للتوسع**: متوسطة (6/10) ويمكن رفعها بسرعة بعد تنظيف lint والهيكلة.
- **الاستقرار طويل المدى**: يحتاج عمل منهجي في typing/testing (5/10).

## خلاصة
المشروع قوي من ناحية الاختيارات التقنية والقدرة على بناء منتج بسرعة، لكنه يحتاج **Sprint تحسين جودة** واضح قبل إضافة ميزات كبيرة جديدة. الاستثمار في التنظيف الآن سيخفض تكلفة التطوير لاحقًا بشكل ملحوظ.
